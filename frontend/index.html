Understanding the data model

Understanding the Unified Data Model

Same idea as the RAG deck — but for data. One clean flow turns any messy file into a business model an AI can answer from. The whole thing is an ontology — a plain map of the business: the things you track, the facts about them, and how they connect. Four moves: land → map → govern → reason.

① Land any file, untouched ② Map columns → business objects ③ Govern health · lineage · access ④ Reason the agent answers
The building blocks

The pieces, in plain terms

Object

A business thing you care about — Company, Product, PricePoint. Like a noun.

Property

A field on an object — and a pointer to a raw column. Product.tds points at t_specs.tds.

Binding

The pointer + a clean rule (strip ₹, ×10 crore→mn). All the intelligence lives here, not in the data.

Link

A relationship between objects — Product competesWith Product. The part a flat table can't do.

Resolver

Turns a business question into real SQL over the raw tables — on the fly, from the bindings.

Raw vs Meaning

Raw data stays raw. The meaning layer holds no data — only pointers. Two layers; that's the trick.

① Zoom in — what an object really is

An object is a business thing (a noun) with a set of properties. Each property doesn't store a value — it points at a column in a raw file. The object itself holds no data; it's a clean set of labelled pointers.

② What a pointer (binding) is — a contacts list for your data

You never dial a raw phone number — you tap a name and the phone looks it up. A pointer works the same way: the agent asks for Product.tds; the resolver looks up the pointer and reads the real column. The pointer also carries the clean rule (strip ₹, convert crore→mn) — that's where all the intelligence lives.

③ What a link is — relationships a flat table can't hold

Objects connect to each other through links — Brand sells Product, Product hasPrice PricePoint, Product competesWith Product. This graph is what lets the agent reason relationally (“who competes with Bisleri on 1L?”), which a single flat table cannot.

④ How the agent answers — fetching across many objects at once

This is the key part. When you ask about a company or scenario, the agent does not read one giant table. It works out which objects hold a slice of the answer, asks each one, the resolver turns every ask into real SQL over the right raw file(s), and then it fuses the numbers and the narrative into a single cited answer.

Step by step: ① the agent reads the question and decides which objects hold a slice of it. ② it asks each object only what it can answer. ③ the resolver turns every ask into real SQL over the right raw file, joining across objects on shared keys (brand, sku) and the links. ④ clean rows + retrieved narrative come back, and the agent fuses them into one cited answer. The same machinery runs for any company or scenario — only the set of objects asked changes.

↑ That's exactly what the narrated walkthrough at the top of this page shows in motion — the question fanning out to each object, the pointers fetching from raw, and the answer fusing together. None of it is pre-stored; it's the path every question takes, every time.

The punchline: the database stays stable on purpose. A new, differently-shaped file doesn't change the concepts — it just adds a few pointers underneath them. Real Python + SQLite; the SQL you'll see on the Agent tab is generated from the bindings and actually run. Nothing here is faked.

Compared to the other approaches

How it relates to the two proposals — RAG vs routing tables

Pure RAG turns every document into text and finds answers by similarity. That's great for narrative, but weak at exact numbers, can't really add things up, gets noisier as more domains are mixed in, and it's hard to audit why an answer appeared. This model uses real, validated queries for the numbers and keeps text-search only for the words — so you get precise figures and the narrative, without the noise.

Yes — it's that approach, generalised and made visual. The ontology is the routing registry: each question is routed to the right objects, answered with validated read-only SQL on an approved set of tables, with narrative search scoped alongside. It's the recommended Proposal 2, the same pattern already live for ESG and PEL — just presented as one clean model.

No. The agent never writes free-form SQL. The system generates read-only (SELECT) queries against an approved allow-list of tables — no edits, no deletes — and every read is validated and logged. The AI decides what to ask; it doesn't get raw access to the database.

Numbers come from querying the actual data, not from matching the closest-looking passage. Similarity search is used only to find relevant words — never to produce a figure. So a benchmark or a total is computed from the real values, every time.

One big vector store blurs domains together and degrades as you add more, can't do exact math, and makes "why this answer" hard to trace. Keeping facts in a queryable model and using search only for text gives precision that holds up as the domains grow.

Mostly data, not code: register the new domain's objects and pointers and it's live. The same safety guardrails (read-only + allow-list) and the existing test suite apply to it automatically — so a new domain inherits the protections for free.

Same core idea — a governed map of meaning over messy data that an AI can reason on. The difference is it's ours, built for our use cases, far faster and cheaper to stand up than a full Foundry deployment, and ours to extend.
